Getting started

Kiln keeps one knowledge graph linking intent → requirement → blueprint → work order, and hands coding agents fully-contextualized work orders over MCP. There are two ways in, depending on whether you’re starting fresh or already have a codebase.

Greenfield

Starting from scratch

No code yet. You author the graph top-down — intent first — and let agents draft the details behind accept/reject gates.

Brownfield

You already have a repo

Bootstrap the graph from the code. A survey agent reads the repository and proposes the feature tree as gated suggestions you review.

The mental model

Everything in Kiln is one of four linked document types, arranged as a tree under a single product root:

  • Requirement — a capability the user gains. The root requirement is your PRD; each feature is a requirement child_of the root.
  • Blueprint — the technical design for a requirement (its details).
  • Work order — one implementable unit, linked back to the blueprint that designed it (implements).
  • Artifact — source material and evidence a requirement is built from (references).

The point of the links is context assembly: when an agent pulls a work order, it receives the whole chain — work order → blueprint → requirement → root intent, plus referenced artifacts — in a single call. Authoring is always top-down (intent before detail), and every AI-generated change lands as a suggestion you accept or reject. Nothing is silently rewritten.

Two rules worth knowing up front. A new capability becomes a new feature requirement under the root. Everything else — a bug fix, refactor, polish — is a work order under an existing feature, never a new requirement. And drafts are allowed to be messy; completeness is only checked at the draft → ready boundary.

Prerequisites

The desktop app is the only install Kiln requires:

  • Author and review a graph — just the desktop app. Download it from the home page (macOS / Windows). Add an Anthropic API key in Settings to enable the authoring agents (draft, refine, review, extract).
  • Hand work orders to a coding agent, or survey an existing repo — the installed app hosts the MCP endpoint itself. Flip Settings → Agent access on, and it generates the token, picks the port, and hands you a copy-paste snippet. No repo clone, no package manager, no terminal — see Handing work to a coding agent.
One store, two surfaces. The human works in the desktop app; a coding agent connects to the app’s bundled MCP endpoint. Both act on the same project files (one SQLite database per project, listed in ~/.kiln/projects.json). The endpoint pins the project that was active when you enabled it — switching the app’s active project never yanks a connected agent. Re-pointing it at another project is an explicit click in Settings.
Headless / CI. If you have no app — a CI runner, a headless box — the same bridge still runs as a standalone server from a repo checkout (git clone, Node 22.5+, pnpm). That path lives in docs/CONNECTING.md → Headless / CI; everyone who downloaded the app should use the toggle instead.

Path A — Greenfield (new project)

You have an idea and no code yet. Author the graph top-down, then generate work orders.

  1. Create the project app

    In the desktop app, open the project switcher (top bar) → New project… → name it. This creates a fully isolated workspace and seeds two documents: a product-root requirement (titled with your project name) and its system architecture blueprint, both empty and waiting for you.

    Prefer the terminal? kiln projects create <name> does the same thing.

  2. Write the product overview app

    Open the root requirement in Documents and describe what the product is, who it’s for, its core capabilities, and — importantly — its non-goals. Fill in the architecture blueprint with your stack and conventions. This is the context every feature will inherit automatically, so it’s worth doing well.

  3. Add features, one capability each app

    For each capability, create a feature requirement child_of the root. Title it <Name> — <plain-language description> (e.g. “X-ray — intent-to-execution map”). Give each a Capability / Why / Scope / Non-goals / Success criteria. You can hand the draft agent your rough notes or an artifact and let it propose the structured requirement and blueprint — you accept or reject.

  4. Cut work orders from a blueprint agents / CLI

    Once a feature’s blueprint is solid, extract implementable work orders from it. The extract agent reads the blueprint and proposes candidate work orders that implement it:

    kiln extract <blueprintId> --accept-all
# or review candidates first, then: kiln accept <id>

    Each work order is born linked to its blueprint, so it inherits the full lineage. Move one to ready when it’s complete (the draft → ready gate checks for the required sections).

  5. Hand it off app MCP

    Turn on Settings → Agent access, paste the snippet into your coding agent, and let it work the board — see Handing work to a coding agent below.

Path B — Brownfield (existing repo)

You already have a codebase and want Kiln to become its living source of truth. Instead of authoring the graph by hand, you run a survey: an agent with filesystem access reads the repo and proposes the product overview and feature tree. Kiln never reads your repository — the agent does, and every proposal lands as a gated suggestion you accept in the app.

  1. Create a fresh, empty project app

    Same as greenfield step 1 — New project… in the app, or kiln projects create <name>. Leave it untouched: the survey only proposes into a fresh project (just the seeded root pair, nothing else). Don’t author anything in it first.

  2. Enable Agent access, pinned to that project app

    With the fresh project active, open Settings → Agent access and flip it on. The app starts a local MCP endpoint pinned to that project, generates a bearer token, and shows the served-project name in its status line — confirm it reads the fresh project before going further. A survey lands dozens of entities; pointed at the wrong project it pollutes a real one.

  3. Connect your coding agent and run the survey skill MCP

    Copy the registration snippet from Settings → Agent access (Claude Code first, then a generic JSON block) and paste it into your MCP client — for the full walkthrough see docs/CONNECTING.md. Install the kiln-survey skill, then ask the agent to “survey this repo into Kiln.” Under the hood it:

    • calls get_project_shape to confirm the target is fresh;
    • proposes the product overview + architecture with propose_root_overview;
    • proposes each capability it finds with propose_feature — each carrying evidence (real file paths and excerpts it can cite).

    These three are the only document writes that exist over MCP, and both propose calls are gated: they create empty-bodied documents plus a pending suggestion per document. Nothing is committed by the agent.

  4. Review and accept in the app app

    Open the project in the desktop app. Every proposed overview, feature, and blueprint is waiting in the suggestion / proposal queue. Read them against the evidence, then accept or reject each one. What you accept becomes the real graph; the rest is discarded. This gate is the whole point — the survey drafts, you decide.

  5. Cut work orders and hand off agents / MCP

    From here it’s identical to greenfield: extract work orders from the accepted blueprints and hand them to a coding agent. Your existing code now has a traceable graph above it.

Handing work to a coding agent

Turn on Settings → Agent access in the app. It starts a local MCP endpoint pinned to the active project, generates a bearer token, and renders a copy-paste registration snippet — the Claude Code command first, then a generic JSON config block. Paste it into your agent and you’re connected; no terminal, no repo checkout.

From there any MCP client (Claude Code, Cursor, …) works the board through three tools: list_ready_work_orders → get_work_order → update_work_order_status. The agent installs the kiln-execute skill, picks a ready work order, receives the full lineage in one payload (and Kiln records a context receipt on delivery), implements it, and reports status back. Execution agents never author or edit Kiln documents — that only happens through suggestions a human resolves.

The full connection walkthrough — the toggle, the snippet, and the loop, plus the standalone headless / CI server — is in docs/CONNECTING.md.

Which surface does what

Kiln has three doors into the same per-project store. Knowing which does what saves confusion:

  • Desktop app app — the human surface. Create projects, author documents, review and accept suggestions, and see the graph (Pulse dashboard, Board, X-ray map, Context Inspector). This is what you download.
  • CLI terminal — scriptable authoring: create, link, draft, extract, review, status, export, projects. Same store as the app.
  • MCP endpoint app MCP — the agent surface. The execution loop plus the gated survey tools. The desktop app hosts it behind the Agent access toggle, pinned to one project; a standalone server serves the same bridge for headless / CI.
Everything stays local and portable. Each project is one SQLite file on your machine — your specs, your key, your data. kiln export <dir> writes the whole graph out as markdown whenever you want it in plain files.
